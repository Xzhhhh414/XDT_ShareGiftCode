# XDT Share Gift Code

玩家向游戏兑换码聚合工具。当前目标是聚合网上玩家公开分享的《心动小镇》兑换码，让玩家快速查看、判断、复制兑换码，并回到游戏内领取奖励。

## 项目定位

这是一个玩家向工具，不是官方兑换码生成或发放系统。

工具只聚合公开来源中的兑换码信息，不自行生成兑换码。来源链接应在数据层保留，供内部审核、去重和追溯使用，但前台不展示给玩家。

## 核心用户价值

- 玩家可以快速看到当前可能可用的兑换码。
- 玩家可以一键复制兑换码，减少手动输入成本。
- 玩家可以判断兑换码是否还可用、是否即将过期。
- 玩家可以反馈兑换码可用或失效，帮助后续排序和隐藏判断。

## 数据来源范围

第一阶段限定两个来源：

- TapTap：优先支持，适合自动或半自动采集公开帖子、动态、攻略内容。
- 小红书：可作为补充来源，但反爬、登录态和平台规则更复杂，初期建议半自动处理。

示例来源：

- https://www.taptap.cn/moment/823255749198089869

## 数据原则

- 只展示兑换码、必要摘要、有效期、验证状态等工具信息。
- 不大段搬运原帖正文或图片。
- 不绕过登录、不抓取私密内容、不做高频请求。
- 同一个兑换码可能来自多个来源，应合并为一条记录。
- 未确认有效期时不要伪造确定状态，前台展示缺失明确有效期和最新玩家反馈。

## 工具方案设定

### 工具形态

- 当前先做静态 HTML 工具，验证页面体验和信息结构。
- 使用同一套数据和组件，通过响应式布局适配 PC Web 和 Mobile Web。
- 页面首屏直接展示兑换码工具，不做营销落地页。
- 后续如果需要跨玩家共享反馈和采集结果，再引入服务器或等价后端存储。

### 前台状态

前台不再在兑换码卡片顶部展示 `可用` / `待验证` 标签。

说明：

- 有明确有效期且尚未过期的兑换码，直接视为当前可用。
- 无法确认有效期的兑换码，展示缺失明确有效期和最新玩家反馈，并允许玩家反馈 `可用` 或 `失效`。
- 确定过期或确定失效超过隐藏缓冲期的兑换码不展示给玩家。

内部数据可以保留更细状态，用于采集、审核、排序和隐藏。

### 有效期展示

前台有效期统一带标题展示：

```text
有效期：今日 24:00 前
有效期：7月31日 24:00 前
有效期：7月7日 20:00 前
```

有效期值只使用以下写法：

```text
今日 24:00 前
7月31日 24:00 前
7月7日 20:00 前
```

规则：

- 来源写“今日有效”“当天 24 点前”等，统一展示为 `今日 24:00 前`。
- 来源给出日期但没有具体时间，统一展示为 `M月D日 24:00 前`。
- 来源给出具体时间，统一展示为 `M月D日 HH:mm 前`。
- 无法从来源确认有效期时，不展示 `有效期：有效期不确定`，改为展示缺失明确有效期和最新玩家反馈。
- 已过期兑换码不进入玩家可见列表。

### 反馈与数据存储

- 玩家可以对兑换码反馈 `可用` 或 `失效`。
- 有明确有效期的兑换码，不展示玩家兑换成功或失败。
- 无明确有效期的兑换码，不展示可用 / 失效累计数量，只展示最新一次玩家反馈结果和时间。
- 无明确有效期的兑换码，卡片内展示单个 `欢迎分享使用结果` 按钮。
- 点击 `欢迎分享使用结果` 后打开弹窗，弹窗内用感谢文案说明玩家反馈价值，并由玩家选择 `可用` 或 `失效`。
- 最新反馈为 `可用` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换成功`。
- 最新反馈为 `失效` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换失败，已失效`。
- 通过玩家反馈判定失效的未知有效期兑换码，不会立刻从前台消失；最新失效反馈超过 12 小时后隐藏。
- 静态 MVP 阶段，反馈可以先存在浏览器本地，用于验证交互。
- 真实服务玩家时，需要后端存储兑换码、来源、玩家提交、玩家反馈和审核结果。

## 推荐产品结构

整体采用同一套数据和组件，通过响应式布局适配 PC Web 和 Mobile Web。不要维护两套完全独立页面。

### PC Web

PC 版强调最新兑换码列表和头图下方的兑换方法说明。

布局建议：

```text
顶部：
心动小镇兑换码 / 最近更新时间

头图下方：
兑换码使用方法：点击手表内“设置” - 点击底部“兑换码”入口 - 输入兑换码即可

主列表：
最新兑换码卡片列表，大屏三列展示
默认按发布时间倒序
```

### Mobile Web

Mobile 版强调领取动作和单手操作，不展示筛选条和统计模块。

布局建议：

```text
顶部 sticky：
心动小镇兑换码
最近更新时间

主体：
兑换码使用方法
单列兑换码卡片
兑换码大字展示
复制按钮作为主操作
有效期 / 最新玩家反馈
```

## 兑换码卡片信息

每条兑换码建议包含：

- 语义标题：由来源上下文、奖励内容、有效期或玩家反馈生成，不直接用兑换码本身做标题；视觉上应区别于兑换码文本
- 兑换码文本：使用更接近可复制代码的样式展示，尺寸和字重应低于语义标题
- 兑换码右侧的一键复制按钮
- 有效期：明确有效期使用统一文案；未知则显示缺失明确有效期和最新玩家反馈
- 奖励内容：如果来源中明确提到则展示，否则不强填
- 无明确有效期时展示最新一次玩家反馈结果和时间
- 用户反馈入口：卡片内为 `欢迎分享使用结果`，弹窗内选择 `可用` 或 `失效`

## 数据模型草案

前端可以先读取静态 JSON，后续再替换为接口。

```json
{
  "code": "TY9H5AK",
  "title": "今日限时礼包码",
  "game": "心动小镇",
  "status": "active",
  "reward": "奖励待确认",
  "expireAt": "2026-07-06T23:59:59+08:00",
  "sourcePlatform": "taptap",
  "sourceUrl": "https://www.taptap.cn/moment/823255749198089869",
  "firstSeenAt": "2026-07-06T10:00:00+08:00",
  "lastSeenAt": "2026-07-06T10:00:00+08:00",
  "sourceCount": 1,
  "latestFeedback": {
    "result": "valid",
    "at": "2026-07-06T10:30:00+08:00"
  },
  "copiedCount": 128
}
```

状态字段建议：

```text
active      内部使用，有明确有效期且未过期
unknown     内部使用，有效期不确定
expired     内部使用，前台隐藏
```

标题生成建议：

- 优先使用语义解析生成的 `title` 字段。
- 来源明确奖励时，标题可围绕奖励生成，如 `家具材料礼包码`。
- 来源明确当天或短期有效时，标题可围绕时效生成，如 `今日限时礼包码`。
- 奖励和有效期都不明确时，不要直接用兑换码做标题，使用 `待验证礼包兑换码` 等兜底文案。

## 信息排序建议

默认排序：

1. 按发布时间倒序展示，最新发布的兑换码排在最前。
2. 确定过期或确定失效的兑换码不展示。

如果后续数据里有多个时间字段，排序优先级建议：

- 来源发布时间
- 首次发现时间
- 最近发现时间

## 采集与处理流程

第一阶段可以先采用手动或半自动数据维护，尽快验证页面体验。

推荐演进路径：

1. 手动维护 `codes.json`
2. 页面支持展示、复制
3. 加入可用 / 失效反馈入口
4. TapTap 来源做半自动采集和人工审核
5. 小红书来源支持人工提交链接和审核
6. 后续再做自动去重和失效检测

抽取兑换码时，可以优先识别：

- 英文数字组合
- 兑换码、礼包码、今日码、领取、有效期等附近文本
- 明确出现“当天 24 点前”“截止到某日”等过期描述

需要排除：

- URL
- 用户 ID
- 日期
- 话题编号
- 普通商品或攻略编号

## 界面参考方向

不建议做成攻略文章页，也不建议做成论坛信息流。更适合做成“兑换码任务清单”。

参考方向：

- 优惠码聚合网站的卡片流：强调“复制 / 验证 / 过期”
- 游戏 Wiki 的兑换码表格：强调“Code / Reward / Notes”的信息清晰度
- 游戏福利聚合站：强调“今日可领 / 剩余时间 / 热度”

最终视觉方向：

- PC：卡片流为主，大屏三列，窄桌面和平板可降为双列
- Mobile：单列卡片，大号复制按钮
- 语气：玩家向、轻工具、不过度官方

## MVP 范围

第一版建议只包含：

- 心动小镇兑换码列表
- PC / Mobile 响应式布局
- 一键复制
- 有效期展示
- 玩家反馈：可用 / 失效
- 静态 JSON 数据源

后续阶段计划见 [docs/PLAN.md](docs/PLAN.md)。

暂不做：

- 完全自动爬虫
- 多游戏支持
- 用户登录体系
- 积分、排行或复杂社区功能
- 自动兑换或模拟游戏客户端行为

## 开发注意事项

- 修改任何内容时，需要同步检查并更新产品文档。当前产品文档入口是本文档和 [docs/PLAN.md](docs/PLAN.md)。
- 如果修改只涉及纯格式化、注释调整或不改变用户可见行为的内部整理，可以不更新产品文档，但交付说明中需要写明原因。
- 页面首屏应直接是工具本身，不做营销落地页。
- 移动端复制按钮必须足够明显，避免用户需要长按选择文本。
- 兑换码文本应保持可选中，复制失败时也能手动复制。
- 已失效或已过期兑换码不展示给玩家，但数据可在后台或数据文件中保留用于去重和追溯。
- 来源链接保留在数据层用于内部追溯，不在前台展示给玩家。
- 所有时间统一按 Asia/Shanghai 展示。
