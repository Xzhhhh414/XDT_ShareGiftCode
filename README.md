# XDT Share Gift Code

玩家向游戏兑换码聚合工具。Phase 1 已完成静态 HTML MVP，Phase 2 已完成本地 Node API 和最小后台原型，Phase 3 已开始支持本机 Codex 采集解析、管理员导入上架和后台下架纠错，用于聚合网上玩家公开分享的《心动小镇》兑换码，让玩家快速查看、判断、复制兑换码，并回到游戏内领取奖励。

## 项目定位

这是一个玩家向工具，不是官方兑换码生成或发放系统。

工具只聚合公开来源中的兑换码信息，不自行生成兑换码。来源链接应在数据层保留，供内部管理、去重和追溯使用，但前台不展示给玩家。

## 核心用户价值

- 玩家可以快速看到当前可能可用的兑换码。
- 玩家可以一键复制兑换码，减少手动输入成本。
- 玩家可以判断兑换码是否还可用、是否即将过期。
- 玩家可以反馈兑换码可用或失效，帮助后续排序和隐藏判断。
- 玩家可以补充待确认的礼包奖励内容，帮助其他玩家判断是否需要兑换。

## 数据来源范围

第一阶段限定两个来源：

- TapTap：优先支持，适合自动或半自动采集公开帖子、动态、攻略内容。
- 小红书：可作为补充来源，但反爬、登录态和平台规则更复杂，初期建议半自动处理。

示例来源：

- https://www.taptap.cn/moment/823255749198089869

## 数据原则

- 只展示兑换码、必要摘要、有效期、验证状态等工具信息。
- 不大段搬运原帖正文或图片。
- 不绕过登录、不抓取私密内容、不做高频请求。
- 同一个兑换码可能来自多个来源，应合并为一条记录。
- 未确认有效期时不要伪造确定状态，前台展示缺失明确有效期和最新玩家反馈。

## 工具方案设定

### 工具形态

- Phase 1 先做静态 HTML 工具，验证页面体验和信息结构，目前已完成。
- Phase 2 已完成本地 Node 服务原型，一个服务同时托管前端静态文件和 API，供本地开发和联调。
- 使用同一套数据和组件，通过响应式布局适配 PC Web 和 Mobile Web。
- 页面首屏直接展示兑换码工具，不做营销落地页。
- 正式发布时，玩家前端不单独部署：将公开 GitHub 仓库 Release 的构建产物 ZIP 导入 TapTap 第三方工具，由 TapTap 托管并展示。
- 腾讯云服务器只承载公开 HTTPS API、兑换码数据和受保护的管理后台；不承载玩家工具的静态页面。
- 玩家工具调用 API 时使用可配置的绝对 HTTPS 地址，服务器仅对白名单中的 TapTap 工具域名开放跨域访问；后台与 API 保持同域。
- Release ZIP 必须是构建产物而非源码，解压后目录含 `index.html`，且资源使用相对路径。
- 初期生产数据使用 SQLite 替换本地 JSON 文件，数据库文件不对公网开放；服务器内存不足或并发需求增长后再迁移到独立数据库服务。

### 前台状态

前台不再在兑换码卡片顶部展示 `可用` / `待验证` 标签。

说明：

- 有明确有效期且尚未过期的兑换码，直接视为当前可用。
- 无法确认有效期的兑换码，展示缺失明确有效期和最新玩家反馈，并允许玩家反馈 `可用` 或 `失效`。
- 确定过期或确定失效超过隐藏缓冲期的兑换码不展示给玩家。

内部数据可以保留更细状态，用于采集、审核、排序和隐藏。

### 有效期展示

前台有效期统一带标题展示：

```text
有效期：今日 24:00 前
有效期：7月31日 24:00 前
有效期：7月7日 20:00 前
```

有效期值只使用以下写法：

```text
今日 24:00 前
7月31日 24:00 前
7月7日 20:00 前
```

规则：

- 来源写“今日有效”“当天 24 点前”等，统一展示为 `今日 24:00 前`。
- 来源给出日期但没有具体时间，统一展示为 `M月D日 24:00 前`。
- 来源给出具体时间，统一展示为 `M月D日 HH:mm 前`。
- 无法从来源确认有效期时，不展示 `有效期：有效期不确定`，改为展示缺失明确有效期和最新玩家反馈。
- 已过期兑换码不进入玩家可见列表。

### 反馈与数据存储

- 玩家可以对兑换码反馈 `可用` 或 `失效`。
- 有明确有效期的兑换码，不展示玩家兑换成功或失败。
- 无明确有效期的兑换码，不展示可用 / 失效累计数量，只展示最新一次玩家反馈结果和时间。
- 无明确有效期的兑换码，卡片内展示 `分享使用结果` 按钮。
- 点击 `分享使用结果` 后打开弹窗，弹窗内用感谢文案说明玩家反馈价值，并由玩家选择 `可用` 或 `失效`。
- 英文数字码与公开来源明确标注的中文口令，都可以提交使用结果反馈。
- 最新反馈为 `可用` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换成功`。
- 最新反馈为 `失效` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换失败，已失效`。
- 未知有效期兑换码收到任一玩家的 `失效` 反馈后立即下架；后台保留反馈、下架时间和原因，补齐奖励与有效期后可重新上架。
- 静态 MVP 阶段，反馈可以先存在浏览器本地，用于验证交互。
- Phase 2 本地原型通过 Node API 写入 `server/db.json`，用于验证跨浏览器共享数据闭环。
- 真实服务玩家时，需要把本地 JSON 存储替换为 VPS 或公司服务上的持久化数据库，并补充生产鉴权、备份、日志和运维能力。

### 奖励内容补充

- 来源明确奖励内容时，卡片直接展示奖励内容。
- 来源未明确奖励内容时，统一展示 `奖励待确认`。
- 奖励为 `奖励待确认` 的卡片，展示 `分享奖励内容` 按钮。
- 点击 `分享奖励内容` 后打开弹窗，弹窗内用感谢文案说明玩家补充价值，并由玩家填写奖励内容。
- 没有账号体系时，`分享奖励内容` 不因为某次提交而消失；玩家可以随时再次补充或修正奖励内容。
- 同一张卡片同时存在奖励补充和使用结果反馈时，两个按钮在卡片底部并排展示。
- 静态 MVP 阶段，玩家补充的奖励内容先存在浏览器本地，用于验证交互；真实服务玩家时，需要后端存储、审核并分发给其他玩家。
- Phase 2 本地原型会把奖励补充写入服务端 JSON，并进入后台待审核；审核通过后才会替换前台展示的奖励内容。

## 本地开发

Phase 2 使用无外部依赖的 Node 服务，本地一个命令即可同时启动前端页面和 API。

```bash
npm start
```

启动后访问：

```text
http://localhost:4173
```

可用脚本：

```bash
npm run check
```

本地数据文件：

- `server/db.seed.json`：仓库内置的初始数据。
- `server/db.json`：本地运行时数据，首次启动时由 seed 自动生成，并被 `.gitignore` 忽略。
- 服务端 JSON 写入使用串行队列，适合本地原型验证；正式上线前仍建议替换为数据库。
- 公网部署前必须设置 `ADMIN_PASSWORD`。服务端会从该密码派生会话签名密钥，不提交任何密钥到仓库。

本地启动后台的示例：

```powershell
$env:ADMIN_PASSWORD = "替换为高强度管理员密码"
npm start
```

当前 API：

```text
GET  /api/health
GET  /api/gift-codes
GET  /api/admin/session
POST /api/admin/login
POST /api/admin/logout
GET  /api/admin/overview
POST /api/admin/import-preview
POST /api/admin/import-publish
POST /api/admin/gift-codes/:code/takedown
POST /api/admin/gift-codes/:code/restore
DELETE /api/admin/gift-codes/:code
POST /api/gift-codes/:code/feedback
POST /api/gift-codes/:code/reward-feedback
```

前端优先读取 `/api/gift-codes`。如果 API 不可用，则回退到 `data/codes.js`，保持直接打开 `index.html` 时仍可查看静态页面。

### TapTap Release 包

本地调试仍使用根目录的 `index.html` 和同源 API。发布到 TapTap 时，不上传后台、服务端源码或运行时数据，而是生成单独的玩家工具 ZIP：

```powershell
# 仅生成目录，用于检查构建产物
npm run build:taptap

# 公网 IP 的 HTTPS 证书验证通过后，生成可上传到 GitHub Release 的 ZIP
$env:TAPTAP_API_BASE_URL = "https://124.222.121.67"
npm run package:taptap
```

输出文件是 `dist/xdt-share-gift-code.zip`。ZIP 内只包含一个 `xdt-share-gift-code` 目录，目录内含 `index.html`、玩家脚本、样式、资源和运行时 API 配置。将该 ZIP 作为公开 GitHub Release 的附件，再从 TapTap 第三方工具页面选择对应 Release。

`TAPTAP_API_BASE_URL` 是服务器 HTTPS 根地址，不包含 `/api`。当前 IP 直连试验使用 `https://124.222.121.67`，会让玩家工具请求 `https://124.222.121.67/api/gift-codes`。该 IP 使用 Let's Encrypt `shortlived` 证书，证书有效期约 6 天，由 acme.sh 自动续期并 reload Caddy；只有浏览器信任该证书后，才能在 TapTap HTTPS 页面使用。

服务器正式启动时使用：

```text
SERVE_PLAYER_STATIC=false
PLAYER_CORS_ORIGINS=https://3rd-tool-h5-al.tapimg.com
ADMIN_PASSWORD=管理员密码
```

`PLAYER_CORS_ORIGINS` 只填写实际工具页面的 Origin，多个域名以英文逗号分隔；不能使用 `*`。2026-07-15 当前工具的实际 Origin 为 `https://3rd-tool-h5-al.tapimg.com`，不能误填工具详情页地址 `https://www.taptap.cn`。管理员后台与 API 同域，不需要加入此变量。

生产服务使用 SQLite 时再设置：

```text
DATA_STORE=sqlite
SQLITE_DB_PATH=/var/lib/xdt-share-gift-code/db.sqlite
```

首次以 `DATA_STORE=sqlite` 启动时，服务会优先导入已有的 `server/db.json`；文件不存在时导入 `server/db.seed.json`。后续所有读写都进入 SQLite 的 WAL 数据库，原始 JSON 不再参与运行时写入。

服务器运行配置、SQLite 数据迁移与 IP HTTPS 验证步骤见 [deploy/README.md](deploy/README.md)。

本地后台页面：

```text
http://localhost:4173/admin.html
```

后台必须先登录才能访问。登录页用两段式选项卡选择身份，默认选中`热心镇民`：完整`管理员`使用服务器配置的 `ADMIN_PASSWORD`，可访问五个后台分页；`热心镇民`使用固定密码 `xdt2026`，只能访问`兑换码管理`。权限会由服务端校验，不能仅靠前端页面绕过。

完整管理员用于导入本机 Codex 采集结果、管理已上架兑换码、处理历史待审核提交和奖励补充，并查看使用结果。后台采用五分页结构：`采集导入`、`兑换码管理`、`新兑换码`、`奖励补充`、`使用结果`。热心镇民只看到兑换码管理列表。后台不提供返回前台入口，保持管理工具独立。

`采集导入` 仍可用于人工处理候选。默认采集链路由仓库内 `collect-gift-codes` Codex skill 按公开来源和发布日期范围低频采集帖子：TapTap 分别枚举“兑换码”和“礼包码”关键词搜索的固定会话分页，再按原始发布时间筛选范围；会话变化、重复 ID 或未走到末页时不生成候选或上传。通过验证后会同时生成候选 JSON 和列出关键词帖、解析结果、排除原因的审计 JSON。skill 仅自动上架满足完整覆盖、高置信、HTTPS 来源、证据明确包含兑换码、奖励明确且无歧义的候选；奖励待确认的候选不会上传。服务端仍会校验兑换码形态、来源链接、证据、有效期和重复状态。其余候选只留在审计文件中，不会写入线上。

自动上传使用 `npm run publish:gift-codes -- <候选文件>`，并要求当前 shell 显式设置 `XDT_GIFT_API_BASE_URL` 与 `XDT_GIFT_ADMIN_PASSWORD`。脚本只将奖励明确的高置信候选发送给完整管理员接口，不保存或输出密码。Codex skill 不读取或保存 `OPENAI_API_KEY`。

`兑换码管理` 先按状态、再按更新时间倒序排列，状态顺序为：`待确认`、`展示中`、`已下架`、`已过期`。奖励或有效期缺失的信息会归为待确认，方便管理员优先判断。

下架会写入 `visible: false`、下架时间和下架原因；前台会继续按 `visible: false` 隐藏该兑换码。完整管理员额外可执行永久`删除`：会移除兑换码及其关联的使用反馈、奖励反馈和历史提交，不能恢复；热心镇民没有此权限。

兑换码管理展示 `待确认`、`展示中`、`已下架`、`已过期` 四种状态。已过期兑换码由前台自动隐藏。

新兑换码或奖励补充支持 `通过` / `拒绝`；使用结果反馈只查看，不做审核。

后台排序规则：

- 新兑换码和奖励补充：待审核在前；同一状态内按提交时间倒序。
- 使用结果：按反馈时间倒序。

后台中新兑换码的来源如果是 `http` / `https` 链接，会展示为可点击链接；非链接说明仍按普通文本展示。来源链接在列表里单行省略显示，完整地址保留在跳转链接和悬停提示中。

后台列表列宽按管理优先级分配，不做等宽：状态和操作列更窄，兑换码和时间列中等，名称、奖励、来源等需要阅读的字段更宽。

玩家前台不再提供新兑换码分享入口，`POST /api/submissions` 也不再开放。后台保留历史提交记录，完整管理员可继续完成历史记录的审核；奖励补充审核通过后才会替换前台展示的奖励内容。

## 兑换码卡片信息

每条兑换码建议包含：

- 语义标题：由来源上下文、奖励内容、有效期或玩家反馈生成，不直接用兑换码本身做标题；视觉上应区别于兑换码文本
- 礼包码发布日期：使用 `firstSeenAt` 展示为显眼标签，只显示月日，缺失时可降级使用 `lastSeenAt`
- 兑换码文本：使用更接近可复制代码的轻量样式展示，尺寸和字重应明显低于语义标题
- 兑换码右侧的一键复制按钮：尺寸跟随兑换码文本保持轻量
- 点击复制后，成功或失败提示固定显示在页面中央
- 有效期：明确有效期使用统一文案；未知则显示缺失明确有效期和最新玩家反馈
- 奖励内容：如果来源中明确提到则展示，否则显示 `奖励待确认`，并提供 `分享奖励内容` 入口
- 无明确有效期时展示最新一次玩家反馈结果和时间
- 用户反馈入口：卡片内为 `分享使用结果`，弹窗内选择 `可用` 或 `失效`

## 数据模型草案

前端可以先读取静态数据文件，后续再替换为接口。

```json
{
  "code": "TY9H5AK",
  "title": "今日限时礼包码",
  "game": "心动小镇",
  "status": "active",
  "reward": "奖励待确认",
  "expireAt": "2026-07-08T23:59:59+08:00",
  "sourcePlatform": "taptap",
  "sourceUrl": "https://www.taptap.cn/moment/823255749198089869",
  "firstSeenAt": "2026-07-08T10:00:00+08:00",
  "lastSeenAt": "2026-07-08T10:00:00+08:00",
  "sourceCount": 1,
  "latestFeedback": {
    "result": "valid",
    "at": "2026-07-08T10:30:00+08:00"
  },
  "copiedCount": 128
}
```

状态字段建议：

```text
active      内部使用，有明确有效期且未过期
unknown     内部使用，有效期不确定
expired     内部使用，前台隐藏
```

标题生成建议：

- 优先使用语义解析生成的 `title` 字段。
- 来源明确奖励时，标题可围绕奖励生成，如 `家具材料礼包码`。
- 来源明确当天或短期有效时，标题可围绕时效生成，如 `今日限时礼包码`。
- 奖励和有效期都不明确时，不要直接用兑换码做标题，使用 `待验证礼包兑换码` 等兜底文案。

## 信息排序建议

默认排序：

1. 按发布时间倒序展示，最新发布的兑换码排在最前。
2. 确定过期或确定失效的兑换码不展示。

如果后续数据里有多个时间字段，排序优先级建议：

- 来源发布时间
- 首次发现时间
- 最近发现时间

## 采集与处理流程

第一阶段可以先采用手动或半自动数据维护，尽快验证页面体验。

推荐演进路径：

1. 手动维护静态数据文件
2. 页面支持展示、复制
3. 加入可用 / 失效反馈入口
4. 加入奖励待确认的玩家补充入口
5. 加入本地 Node API 和服务端 JSON 存储
6. TapTap 来源做半自动采集、解析预览和直接上架
7. 小红书来源支持提交公开链接或正文解析，并进入直接上架流程
8. 后续再做自动去重和失效检测

抽取兑换码时，可以优先识别：

- 英文数字组合，或被明确标注为兑换码 / 礼包码的中文口令
- 兑换码、礼包码、今日码、领取、有效期等附近文本
- 明确出现“当天 24 点前”“截止到某日”等过期描述

需要排除：

- URL
- 用户 ID
- 日期
- 话题编号
- 普通商品或攻略编号

## 界面参考方向

不建议做成攻略文章页，也不建议做成论坛信息流。更适合做成“兑换码任务清单”。

参考方向：

- 优惠码聚合网站的卡片流：强调“复制 / 验证 / 过期”
- 游戏 Wiki 的兑换码表格：强调“Code / Reward / Notes”的信息清晰度
- 游戏福利聚合站：强调“今日可领 / 剩余时间 / 热度”

最终视觉方向：

- PC：卡片流为主，大屏三列，窄桌面和平板可降为双列
- Mobile：单列卡片，大号复制按钮
- 语气：玩家向、轻工具、不过度官方

## Phase 1 完结范围

第一版静态 MVP 已包含：

- 心动小镇兑换码列表
- PC / Mobile 响应式布局
- 一键复制
- 有效期展示
- 玩家反馈：可用 / 失效
- 奖励待确认时的玩家补充入口
- 本地后台查看历史待审核数据
- 静态 JS 数据源

后续阶段计划见 [docs/PLAN.md](docs/PLAN.md)，下一阶段重点是后端存储、共享反馈、公开来源解析和下架纠错。

暂不做：

- 完全自动爬虫
- 多游戏支持
- 用户登录体系
- 积分、排行或复杂社区功能
- 自动兑换或模拟游戏客户端行为

## 开发注意事项

- 修改任何内容时，需要同步检查并更新产品文档。当前产品文档入口是本文档和 [docs/PLAN.md](docs/PLAN.md)。
- 如果修改只涉及纯格式化、注释调整或不改变用户可见行为的内部整理，可以不更新产品文档，但交付说明中需要写明原因。
- 页面首屏应直接是工具本身，不做营销落地页。
- 移动端复制按钮必须足够明显，避免用户需要长按选择文本。
- 兑换码文本应保持可选中，复制失败时也能手动复制。
- 已失效或已过期兑换码不展示给玩家，但数据可在后台或数据文件中保留用于去重和追溯。
- 来源链接保留在数据层用于内部追溯，不在前台展示给玩家。
- 所有时间统一按 Asia/Shanghai 展示。
