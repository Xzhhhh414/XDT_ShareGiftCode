# 开发计划

本文档记录《心动小镇》兑换码聚合工具的开发计划。Phase 1 静态 HTML MVP 已完成，Phase 2 本地 Node API 和最小审核后台原型已完成。后续先完善采集、解析和审核等业务能力，VPS / 正式数据库 / 生产部署等生产化工作最后处理。

## 产品规则调整

### 文档同步规则

修改任何产品行为、界面文案、交互流程、数据结构、样式布局、规则判断或开发阶段计划时，必须同步更新产品文档。

当前产品文档入口：

- `README.md`：记录当前产品定位、核心规则、页面结构和 MVP 范围。
- `docs/PLAN.md`：记录阶段计划、已完成进度、后续演进和需要保留的产品决策。

例外情况：

- 纯格式化
- 注释调整
- 不改变用户可见行为的内部整理

使用例外时，需要在交付说明中明确说明无需更新产品文档的原因。

### 状态精简

前台不再在兑换码卡片顶部展示 `可用` / `待验证` 状态标签。

展示规则：

- 有明确有效期且尚未过期的兑换码，直接视为当前可用。
- 无法确认有效期的兑换码，展示缺失明确有效期和最新玩家反馈，并允许玩家反馈 `可用` 或 `失效`。
- “已失效 / 已过期”的兑换码不再展示给玩家，但通过玩家反馈判定失效的未知有效期兑换码会保留 12 小时缓冲。

说明：

- “即将过期”属于有效期提示，不作为独立状态。
- 内部数据可以保留更细状态，用于审核、采集、隐藏和排序。

### 有效期文案统一

前台有效期只使用以下文案格式：

```text
今日 24:00 前
7月31日 24:00 前
7月7日 20:00 前
```

规则：

- 来源写“今日有效”“当天 24 点前”等，统一展示为 `今日 24:00 前`。
- 来源给出日期但没有具体时间，统一展示为 `M月D日 24:00 前`。
- 来源给出具体时间，统一展示为 `M月D日 HH:mm 前`。
- 没有解析到有效期，不展示 `有效期：有效期不确定`，改为展示缺失明确有效期和最新玩家反馈。
- 已过期兑换码不进入玩家可见列表。

### 过期码展示规则

确定过期的兑换码不展示给玩家。

隐藏条件：

- `expireAt < now`
- 来源明确表达“已失效”“不能用了”“过期了”
- 未知有效期兑换码的最新玩家反馈为 `失效`，且反馈时间已经超过 12 小时

隐藏后的数据仍可以保留在后台或数据文件里，用于：

- 避免重复采集
- 避免重复提交
- 追溯来源
- 训练或优化有效期解析规则

### 玩家反馈规则

对于无法通过来源确认有效期或有效性的兑换码，允许玩家反馈：

- `可用`
- `失效`

反馈不应由单个玩家直接决定全局状态。

前台展示规则：

- 不展示可用 / 失效累计数量。
- 有明确有效期的兑换码，不展示玩家兑换成功或失败。
- 只在无明确有效期的兑换码上展示最新一次玩家反馈结果和时间。
- 无明确有效期的兑换码，卡片内展示 `分享使用结果` 按钮。
- 点击 `分享使用结果` 后打开弹窗，弹窗内用感谢文案说明玩家反馈价值，并由玩家选择 `可用` 或 `失效`。
- 同一张卡片同时存在奖励补充和使用结果反馈时，两个按钮在卡片底部并排展示。
- 没有账号体系时，奖励待确认卡片的 `分享奖励内容` 按钮需要持续展示，允许玩家随时再次补充或修正奖励内容。
- 最新反馈为 `可用` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换成功`。
- 最新反馈为 `失效` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换失败，已失效`。
- 通过玩家反馈判定失效的未知有效期兑换码，不会立刻下架；最新失效反馈超过 12 小时后隐藏。

## 是否需要服务器

静态 MVP 可以不需要服务器，但只能验证页面体验。

如果要让工具真实服务玩家，就需要服务器或等价的后端存储，因为以下信息需要跨玩家共享：

- 兑换码列表
- 玩家提交的新兑换码
- 可用 / 失效反馈
- 玩家补充的奖励内容
- 复制次数
- 来源采集结果
- 人工审核结果
- 过期隐藏判断结果

当前本地 `localStorage` 只能记录单个浏览器的反馈和奖励补充，不具备真实运营价值。

## 阶段计划

### Phase 1：静态 MVP（已完成）

目标：完成可体验的前端工具，验证布局、信息结构和玩家操作流程。

范围：

- 静态 HTML / CSS / JS
- PC / Mobile 响应式布局
- 静态数据文件
- 兑换码卡片展示语义标题，不直接用兑换码本身作为标题
- 兑换码卡片展示礼包码发布日期，并使用显眼标签突出
- 卡片顶部不展示状态标签
- 有效期文案统一
- 过期码前台隐藏
- 默认按发布时间倒序展示
- PC 大屏使用三列兑换码卡片，窄桌面和平板使用双列，Mobile 使用单列卡片
- 一键复制
- 本地反馈交互，未知有效期兑换码通过弹窗发布使用结果
- 本地奖励补充交互，奖励待确认的兑换码通过弹窗填写奖励内容
- 头图下方展示兑换码使用方法说明

不做：

- 后端存储
- 自动采集
- 用户登录
- 真实跨用户反馈统计
- 来源筛选
- 兑换码数量统计模块
- 最新来源面板
- 玩家反馈统计面板
- 提交兑换码入口

### Phase 2：轻量后端与共享数据（本地原型已完成）

目标：让兑换码、反馈、奖励补充和玩家提交新兑换码先成为共享数据；本地后台用于处理待审核数据，为后续采集和语义解析提供完整本地闭环。

当前先做本地开发原型：一个 Node 服务同时托管静态页面和 API，数据写入本地 JSON 文件。这个方案用于验证接口、数据结构和玩家体验，不代表正式上线架构。

范围：

- 增加本地后端 API
- 由同一个 Node 服务托管前端静态文件和 API
- 存储兑换码列表到服务端数据文件
- 存储玩家反馈到服务端数据文件
- 存储玩家补充的奖励内容到服务端数据文件
- 存储玩家提交的新兑换码到服务端数据文件
- 前端从接口读取数据
- 提交和反馈写入接口
- 增加本地后台页面查看待审核新码、奖励补充和使用结果
- 后台采用三分页列表结构：`新兑换码`、`奖励补充`、`使用结果`，每个分页使用表格列表展示
- 后台不提供返回前台入口，保持审核工具独立
- 后台支持待审核新兑换码和奖励补充的 `通过` / `拒绝`
- 使用结果反馈只查看，不进入审核流
- 根据最新玩家反馈同步前台展示和隐藏判断
- 服务端 JSON 写入使用串行队列，避免本地原型并发提交时互相覆盖
- 玩家提交新兑换码时，先检查当前兑换码库存和待审核提交，重复兑换码不能再次提交
- 玩家手动提交新兑换码时，名称、兑换码、奖励、有效期为必填；来源链接或说明为选填
- 新兑换码提交弹窗里，有效期输入区域整体可点击并触发日期选择器
- `分享新的兑换码` 是头图内的次级操作，不在兑换码列表前单独占一整行
- 后台排序规则：新兑换码和奖励补充待审核在前，同状态按提交时间倒序；使用结果按反馈时间倒序
- 后台中新兑换码来源为 `http` / `https` 链接时可点击打开；非链接说明按普通文本展示
- 后台来源链接在列表中单行省略显示，完整地址保留在跳转链接和悬停提示中
- 后台表格列宽按审核优先级分配，不使用等宽列
- 新兑换码审核通过后写入兑换码库存并进入前台列表；奖励补充审核通过后才会替换前台展示的奖励内容

Phase 2 本地原型已完成的边界：

- 登录和账号体系
- 正式数据库
- 自动采集
- 批量审核和复杂权限控制
- 部署到 VPS

这些内容不作为 Phase 2 收口条件。VPS、正式数据库、鉴权、备份、日志和监控等生产化工作，等业务能力完整跑通后再进入部署阶段。

可选增强但不阻塞 Phase 2：

- 增加后台筛选、搜索、审核备注和批量操作
- 增加更细的管理员权限控制

### Phase 3：采集与语义解析

目标：降低人工维护成本，半自动发现新兑换码。

范围：

- TapTap 公开来源采集
- 小红书人工提交链接优先，后续再评估采集方式
- 抽取疑似兑换码
- 解析兑换码附近上下文
- 生成玩家可读的兑换码标题
- 解析有效期
- 识别“可用 / 失效”语义
- 合并同码多来源
- 进入人工审核队列

解析策略：

- 先用规则识别明确日期和时间
- 再用语义模型处理“今日有效”“当天 24 点前”“过期了”等自然语言
- 标题优先从来源上下文、奖励内容、有效期语义生成；信息不足时使用兜底标题，不直接拿兑换码本身当标题
- 不确定时不要强行给过期时间，前台展示缺失明确有效期和最新玩家反馈

### Phase 4：排序信号与自动隐藏

目标：让列表排序和隐藏更接近真实可用情况。

范围：

- 根据信号调整排序，不对玩家展示百分比评分
- 根据最新玩家反馈调整展示和隐藏
- 根据过期时间自动隐藏
- 根据失效反馈阈值隐藏
- 保留隐藏记录用于去重和追溯

排序和隐藏信号：

- 来源数量
- 来源平台
- 来源发布时间
- 最近发现时间
- 最新玩家反馈结果
- 最新玩家反馈时间
- 是否解析到明确有效期
