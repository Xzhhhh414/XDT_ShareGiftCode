# 开发计划

本文档记录《心动小镇》兑换码聚合工具的开发计划。Phase 1 静态 HTML MVP 已完成，Phase 2 本地 Node API 和最小后台原型已完成，Phase 3 已开始支持本机 Codex 采集解析、管理员导入上架和后台下架纠错。正式发布时，玩家前端由 TapTap 第三方工具承载，腾讯云服务器只承载 API、数据和管理后台。

## 产品规则调整

### 文档同步规则

修改任何产品行为、界面文案、交互流程、数据结构、样式布局、规则判断或开发阶段计划时，必须同步更新产品文档。

当前产品文档入口：

- `README.md`：记录当前产品定位、核心规则、页面结构和 MVP 范围。
- `docs/PLAN.md`：记录阶段计划、已完成进度、后续演进和需要保留的产品决策。

例外情况：

- 纯格式化
- 注释调整
- 不改变用户可见行为的内部整理

使用例外时，需要在交付说明中明确说明无需更新产品文档的原因。

### 状态精简

前台不再在兑换码卡片顶部展示 `可用` / `待验证` 状态标签。

展示规则：

- 有明确有效期且尚未过期的兑换码，直接视为当前可用。
- 无法确认有效期的兑换码，展示缺失明确有效期和最新玩家反馈，并允许玩家反馈 `可用` 或 `失效`。
- “已失效 / 已过期”的兑换码不再展示给玩家，但通过玩家反馈判定失效的未知有效期兑换码会保留 12 小时缓冲。

说明：

- “即将过期”属于有效期提示，不作为独立状态。
- 内部数据可以保留更细状态，用于审核、采集、隐藏和排序。

### 有效期文案统一

前台有效期只使用以下文案格式：

```text
今日 24:00 前
7月31日 24:00 前
7月7日 20:00 前
```

规则：

- 来源写“今日有效”“当天 24 点前”等，统一展示为 `今日 24:00 前`。
- 来源给出日期但没有具体时间，统一展示为 `M月D日 24:00 前`。
- 来源给出具体时间，统一展示为 `M月D日 HH:mm 前`。
- 没有解析到有效期，不展示 `有效期：有效期不确定`，改为展示缺失明确有效期和最新玩家反馈。
- 已过期兑换码不进入玩家可见列表。

### 过期码展示规则

确定过期的兑换码不展示给玩家。

隐藏条件：

- `expireAt < now`
- 来源明确表达“已失效”“不能用了”“过期了”
- 未知有效期兑换码的最新玩家反馈为 `失效`，且反馈时间已经超过 12 小时

隐藏后的数据仍可以保留在后台或数据文件里，用于：

- 避免重复采集
- 避免重复提交
- 追溯来源
- 训练或优化有效期解析规则

### 玩家反馈规则

对于无法通过来源确认有效期或有效性的兑换码，允许玩家反馈：

- `可用`
- `失效`

反馈不应由单个玩家直接决定全局状态。

前台展示规则：

- 不展示可用 / 失效累计数量。
- 有明确有效期的兑换码，不展示玩家兑换成功或失败。
- 只在无明确有效期的兑换码上展示最新一次玩家反馈结果和时间。
- 无明确有效期的兑换码，卡片内展示 `分享使用结果` 按钮。
- 点击 `分享使用结果` 后打开弹窗，弹窗内用感谢文案说明玩家反馈价值，并由玩家选择 `可用` 或 `失效`。
- 同一张卡片同时存在奖励补充和使用结果反馈时，两个按钮在卡片底部并排展示。
- 没有账号体系时，奖励待确认卡片的 `分享奖励内容` 按钮需要持续展示，允许玩家随时再次补充或修正奖励内容。
- 最新反馈为 `可用` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换成功`。
- 最新反馈为 `失效` 时，展示 `缺失明确有效期，MM/DD HH:mm 玩家兑换失败，已失效`。
- 通过玩家反馈判定失效的未知有效期兑换码，不会立刻下架；最新失效反馈超过 12 小时后隐藏。

## 是否需要服务器

静态 MVP 可以不需要服务器，但只能验证页面体验。

如果要让工具真实服务玩家，就需要服务器或等价的后端存储，因为以下信息需要跨玩家共享：

- 兑换码列表
- 玩家提交的新兑换码
- 可用 / 失效反馈
- 玩家补充的奖励内容
- 复制次数
- 来源采集结果
- 自动上架结果和后台下架记录
- 过期隐藏判断结果

当前本地 `localStorage` 只能记录单个浏览器的反馈和奖励补充，不具备真实运营价值。

## 正式部署架构

正式环境不单独部署面向玩家的 Web 站点。玩家侧静态页面通过 TapTap 第三方工具发布：公开 GitHub 仓库的每个 Release 提供一个构建产物 ZIP，TapTap 拉取、校验并托管该包，再在工具页中打开。

```text
GitHub Release 构建产物 ZIP
  -> TapTap 第三方工具托管玩家前端
  -> 腾讯云 HTTPS API
  -> 兑换码数据存储

腾讯云 HTTPS 管理后台
  -> 同一份 API 和兑换码数据存储
```

GitHub Release 包要求：

- 仓库公开可访问。
- 上传 `npm run build` 等构建产物的 ZIP，而不是源码压缩包。
- ZIP 解压后只包含一个目录，目录内有入口文件 `index.html`。
- 静态资源使用相对路径，不能假定页面部署在站点根路径。

腾讯云服务器的职责：

- 提供玩家访问的公开 HTTPS API。
- 存储兑换码、来源、玩家反馈、下架记录和后台操作数据。
- 托管受管理员密码保护的后台页面。
- 不托管 TapTap 玩家工具的 `index.html`、`script.js`、`styles.css` 等静态前端文件。

初期正式数据存储使用 SQLite 替换本地 JSON 文件：它与 Node 服务运行在同一台 2 GB 服务器上，数据库文件不开放公网，并通过每日备份保护。数据量、并发或多实例需求增长后再迁移到独立数据库服务。

跨域规则：

- 玩家工具从 TapTap 托管域名调用 API，前端必须使用可配置的绝对 API 地址。
- API 只允许实际 TapTap 工具域名的跨域请求，不使用宽泛的跨域白名单。
- 管理后台与 API 保持同域部署，继续使用同源会话 Cookie。

本地开发仍可由一个 Node 服务同时托管静态页面和 API，便于调试；这不代表正式部署拓扑。

玩家 Release 包由 `npm run package:taptap` 生成。构建时必须提供 `TAPTAP_API_BASE_URL=https://服务器域名`，脚本会输出只包含玩家前端的 `dist/xdt-share-gift-code.zip`；该地址不含 `/api`，玩家脚本会自动请求其 `/api/*` 路径。服务端正式运行时设置 `SERVE_PLAYER_STATIC=false`，禁止同机继续公开玩家静态页面，并用 `PLAYER_CORS_ORIGINS` 仅允许实际 TapTap 工具托管域名跨域访问。

服务器部署以 systemd 管理 Node 服务：服务使用 `xdtgift` 系统用户、仅监听 `127.0.0.1:4173`，SQLite 状态目录为 `/var/lib/xdt-share-gift-code`。备案审核期间可以完成 Node 24 安装、代码拉取、SQLite 初始化和本机回环接口验证；备案通过后才安装 Caddy、开放 `80/443` 并申请 HTTPS 证书。

## 阶段计划

### Phase 1：静态 MVP（已完成）

目标：完成可体验的前端工具，验证布局、信息结构和玩家操作流程。

范围：

- 静态 HTML / CSS / JS
- PC / Mobile 响应式布局
- 静态数据文件
- 兑换码卡片展示语义标题，不直接用兑换码本身作为标题
- 兑换码卡片展示礼包码发布日期，并使用显眼标签突出
- 卡片顶部不展示状态标签
- 有效期文案统一
- 过期码前台隐藏
- 默认按发布时间倒序展示
- PC 大屏使用三列兑换码卡片，窄桌面和平板使用双列，Mobile 使用单列卡片
- 一键复制
- 本地反馈交互，未知有效期兑换码通过弹窗发布使用结果
- 本地奖励补充交互，奖励待确认的兑换码通过弹窗填写奖励内容
- 头图下方展示兑换码使用方法说明

不做：

- 后端存储
- 自动采集

- 来源筛选
- 兑换码数量统计模块

- 玩家反馈统计面板

### Phase 2：轻量后端与共享数据（本地原型已完成）

目标：让兑换码、反馈、奖励补充和玩家提交新兑换码先成为共享数据；本地后台用于处理玩家提交、奖励补充、公开来源解析和下架纠错，为后续采集和语义解析提供完整本地闭环。

当前先做本地开发原型：一个 Node 服务同时托管静态页面和 API，数据写入本地 JSON 文件。这个方案用于验证接口、数据结构和玩家体验，不代表正式上线架构。

范围：

- 增加本地后端 API
- 由同一个 Node 服务托管前端静态文件和 API
- 存储兑换码列表到服务端数据文件
- 存储玩家反馈到服务端数据文件
- 存储玩家补充的奖励内容到服务端数据文件
- 存储玩家提交的新兑换码到服务端数据文件
- 前端从接口读取数据
- 提交和反馈写入接口
- 增加本地后台页面导入公开来源候选文件，管理已上架兑换码，查看待审核新码、奖励补充和使用结果
- 后台采用五分页列表结构：`采集导入`、`兑换码管理`、`新兑换码`、`奖励补充`、`使用结果`，每个分页使用表格列表展示
- 后台不提供返回前台入口，保持管理工具独立
- 后台支持待审核新兑换码和奖励补充的 `通过` / `拒绝`
- 使用结果反馈只查看，不进入审核流
- 根据最新玩家反馈同步前台展示和隐藏判断
- 后台支持对已上架兑换码执行 `下架` / `恢复`，下架后前台不展示
- 兑换码管理只展示 `展示中`、`已过期`、`已下架` 三种状态；未知有效期但仍可展示的兑换码归为 `展示中`，已过期兑换码不提供手动下架操作
- 服务端 JSON 写入使用串行队列，避免本地原型并发提交时互相覆盖
- 玩家提交新兑换码时，先检查当前兑换码库存和待审核提交，重复兑换码不能再次提交
- 玩家手动提交新兑换码时，名称、兑换码、奖励、有效期为必填；来源链接或说明为选填
- 新兑换码提交弹窗里，有效期输入区域整体可点击并触发日期选择器
- `分享新的兑换码` 是头图内的次级操作，不在兑换码列表前单独占一整行
- 后台排序规则：新兑换码和奖励补充待审核在前，同状态按提交时间倒序；使用结果按反馈时间倒序
- 后台中新兑换码来源为 `http` / `https` 链接时可点击打开；非链接说明按普通文本展示
- 后台来源链接在列表中单行省略显示，完整地址保留在跳转链接和悬停提示中
- 后台表格列宽按管理优先级分配，不使用等宽列
- 新兑换码审核通过后写入兑换码库存并进入前台列表；奖励补充审核通过后才会替换前台展示的奖励内容

### Phase 3：采集与语义解析

目标：降低人工维护成本，半自动发现新兑换码。

当前先做本机 Codex 半自动采集，不直接做服务端全自动全站爬虫：仓库内 `collect-gift-codes` skill 接收 TapTap 等公开来源链接和发布日期范围。TapTap 分别枚举“兑换码”和“礼包码”关键词搜索的固定会话分页，再按原始发布时间筛选范围；会话变化、重复 ID 或未走到末页时，skill 不生成可导入候选文件。它只保留标题或正文包含 `兑换码` / `礼包码` 的公开帖子文本，并由当前 Codex 会话完成语义抽取。skill 输出候选 JSON；管理员登录公网后台的 `采集导入` 后读取候选、确认重复状态，再直接上架。服务端不调用 LLM，也不保存 OpenAI Key。

范围：

- TapTap 公开来源采集
- 小红书人工提交链接优先，后续再评估采集方式
- 本机 Codex skill 低频浅层爬取
- 筛出标题或正文包含 `兑换码` / `礼包码` 的公开帖子文本
- Codex 结构化抽取英文数字兑换码或明确标注的中文礼包码
- 生成玩家可读的兑换码标题
- 解析有效期
- 识别“可用 / 失效”语义
- 合并同码多来源
- 直接上架并保留下架纠错能力

已完成的第一步：

- 仓库内新增 `collect-gift-codes` Codex skill，按公开来源和指定发布日期范围采集、筛选与语义解析，生成可导入的候选 JSON 和供规则检查的审计 JSON。
- 后台新增 `采集导入` 分页，读取 Codex skill 生成的候选 JSON 并预览重复状态。
- 新增 `POST /api/admin/import-preview`，用于校验候选文件但不修改数据。
- 新增 `POST /api/admin/import-publish`，用于把未重复候选批量写入兑换码库存并直接上架。
- 新增管理员登录、会话和退出接口；所有 `/api/admin/*` 的管理操作均需会话认证。
- 后台新增 `兑换码管理` 分页，支持对已上架兑换码执行 `下架` / `恢复`。
- 新增 `POST /api/admin/gift-codes/:code/takedown` 和 `POST /api/admin/gift-codes/:code/restore`。
- Codex 只处理包含 `兑换码` / `礼包码` 的文本块，不处理整页 HTML；发布日期不明确或不在范围内的帖子不纳入候选。
- 服务端会校验候选：兑换码可以是 5 到 24 位且同时含字母与数字的英文数字组合，或 2 到 20 个连续汉字且被明确标注为礼包码；不能来自组队链接、URL 参数、用户 ID、日期、图片 hash 或帖子 ID。
- 服务端会过滤已入库兑换码和非拒绝状态的待审核兑换码，避免重复上架或重复提交。
- LLM 会尝试生成标题、奖励内容、有效期和来源摘要；无法确认有效期时保留为空，上架后以前台未知有效期规则展示。
- 当前 TapTap skill 先验证候选历史 feed 的跨页时间顺序、分页快照和日期边界；任一条件无法验证时不输出候选，避免不完整范围误报为最终结果。

解析策略：

- skill 负责抓公开页面、提取同域链接、去掉明显干扰信息，并筛出包含 `兑换码` / `礼包码` 的文本块。
- 当前 Codex 会话负责从文本块中判断是否存在可进入工具的英文数字兑换码或明确标注的中文礼包码，并结构化输出标题、奖励、有效期、置信度和证据。
- 服务端规则负责最终校验、去重、过滤组队码和 URL 参数码，不让本机候选绕过约束。
- 不确定时不要强行给过期时间，前台展示缺失明确有效期和最新玩家反馈。

### Phase 4：排序信号与自动隐藏

目标：让列表排序和隐藏更接近真实可用情况。

范围：

- 根据信号调整排序，不对玩家展示百分比评分
- 根据最新玩家反馈调整展示和隐藏
- 根据过期时间自动隐藏
- 根据失效反馈阈值隐藏
- 保留隐藏记录用于去重和追溯

排序和隐藏信号：

- 来源数量
- 来源平台
- 来源发布时间
- 最近发现时间
- 最新玩家反馈结果
- 最新玩家反馈时间
- 是否解析到明确有效期
